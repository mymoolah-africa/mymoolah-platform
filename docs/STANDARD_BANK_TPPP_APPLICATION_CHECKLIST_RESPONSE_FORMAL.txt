MyMoolah operates under a clear agent appointment framework in compliance with PASA requirements. The platform acts as:


4.1.1 Beneficiary Service Provider (BSP)

When acting as a Beneficiary Service Provider, MyMoolah has been appointed as an agent of each beneficiary (merchant, supplier, biller) through formal agreements.

Proof of Agreements:
- Flash Partner Agreement (Agent of Flash for airtime/data services)
- MobileMart Fulcrum Agreement (Agent of MobileMart for VAS services)
- EasyPay Agreement (Agent of EasyPay for voucher services)
- Zapper Merchant Agreement (Agent of merchants for QR payments)
- DT Mercury Agreement (Agent of DT Mercury for PayShap services)
- Peach Payments Agreement (Agent of Peach Payments for PayShap services)
- Bill Payment Aggregator Agreements (Agent of billers for bill payment services)


4.1.2 Payer Service Provider (PSP)

Proof of Agreements:
- Consumer Terms and Conditions (Digital acceptance on wallet registration)
- B2B Client Agreements (Formal contracts for prefunded float clients)
- Merchant Onboarding Agreements (For merchants accepting MyMoolah payments)

5. RECORD KEEPING REQUIREMENTS

5.1 Record Keeping Compliance

MyMoolah maintains comprehensive records of all payments to third persons in compliance with PASA requirements. All records include:

- Date: Timestamp of the transaction
- Amount: Transaction amount in ZAR
- Beneficiary: Complete beneficiary identification (merchant ID, supplier ID, account number, etc.)
- Transaction Reference: Unique transaction identifier
- Payment Method: Method of payment (RTP, RPP, QR, VAS, etc.)
- Status: Transaction status (completed, failed, pending, etc.)


5.2 Record Retention Period

Retention Period: Five (5) years from transaction date, as required by PASA regulations.


5.3 Technical Implementation

Transaction Record Model:

The platform maintains complete transaction records through the Transaction model, which includes:
- Unique transaction identifier
- Transaction amount (DECIMAL 15,2 for banking-grade precision)
- Transaction type (send, receive, deposit, withdraw, transfer, payment, refund, fee)
- Transaction status (pending, processing, completed, failed, cancelled, reversed)
- Description including beneficiary details
- Metadata in JSONB format for additional beneficiary information
- Created timestamp (retention period starts from this date)

Payment Record Model:

The platform maintains payment records through the Payment model, which includes:
- Payment reference (unique)
- Amount paid in cents
- Payment date (date for retention period calculation)
- Beneficiary merchant identifier
- External transaction ID
- Payment status

Journal Entry Record (Double-Entry Accounting):

The platform maintains immutable audit trail through JournalEntry and JournalLine models:
- Journal entry reference (links to transaction)
- Journal entry description including beneficiary details
- Posted date (retention period starts from this date)
- Journal lines with account identifiers and beneficiary details in memos


5.4 Record Retrieval Capabilities

The platform provides comprehensive transaction history queries with:
- Date range filtering (start date and end date)
- Transaction type filtering
- Status filtering
- Pagination support (limit and offset)
- Related wallet information included


5.5 Audit Trail Implementation

Immutable Journal Posting:

The platform implements atomic journal entry posting with:
- Validation requiring at least two lines
- Validation ensuring debits equal credits (within 0.000001 tolerance)
- Atomic database transactions ensuring data integrity
- Journal lines created with beneficiary details in memos
- Complete rollback on error


5.6 Data Archiving Strategy

The platform implements automated archiving for transactions older than 5 years:
- Archive table structure matches active transaction table
- Annual archiving function maintains referential integrity
- Related journal entries archived simultaneously
- Archived data remains accessible for audit purposes


5.7 Attestation and Audit

6. OPERATIONAL AND TECHNICAL COMPLIANCE

6.1 TPPP Compliance Statement

MyMoolah Treasury Platform operates as a Third-Party Payment Provider (TPPP) in compliance with all PASA TPPP requirements. The platform provides payments to third persons and processes payment instructions through established banking and payment clearing house (PCH) system operator channels.


6.2 Operational Requirements Compliance

6.2.1 Security Standards

TLS 1.3 Implementation:

The platform implements TLS 1.3 with banking-grade security:
- Minimum and maximum version enforced as TLS 1.3
- Strong cipher suites (AES-256-GCM-SHA384, ChaCha20-Poly1305-SHA256, AES-128-GCM-SHA256)
- Perfect Forward Secrecy with ECDHE key exchange
- Certificate validation enabled
- OCSP stapling enabled

Security Headers:

The platform implements comprehensive security headers:
- HTTP Strict Transport Security (HSTS) with 1-year max age
- Content Security Policy (CSP) with strict resource controls
- X-Content-Type-Options: nosniff
- X-Frame-Options: DENY
- X-XSS-Protection: 1; mode=block
- Referrer-Policy: strict-origin-when-cross-origin


6.2.2 Rate Limiting and Input Validation

Rate Limiting:

The platform implements sophisticated rate limiting:
- General API rate limiting: 100 requests per 15 minutes in production
- Financial transaction rate limiting: 10 transactions per minute in production
- Standard rate limit headers included in responses

Input Validation:

The platform implements comprehensive input validation:
- Request validation middleware checks all inputs
- Validation errors returned in structured format
- Field-level error messages provided
- Input sanitization applied before processing


6.2.3 Authentication and Authorization

JWT Authentication:

The platform uses enhanced JWT authentication:
- HS512 algorithm (stronger than HS256)
- Short token expiry: 1 hour in production
- Clock tolerance: 30 seconds for clock skew
- Secure token validation with issuer and audience checks


6.3 Technical Requirements Compliance

6.3.1 Atomic Transaction Processing

Atomic VAS Purchase:

The platform implements atomic transaction processing:
- Database transactions ensure all-or-nothing execution
- Row-level locking prevents concurrent modifications
- Balance validation before transaction execution
- Wallet debit and transaction creation in single atomic operation
- Journal entry posting included in same transaction
- Complete rollback on any error


6.3.2 Double-Entry Accounting

Balanced Journal Entries:

The platform enforces double-entry accounting principles:
- At least two lines required per journal entry
- Debits must equal credits (within 0.000001 tolerance)
- Atomic database transactions ensure integrity
- Validation enforced at application level before database commit


6.3.3 Idempotency

Idempotent Processing:

The platform implements idempotency for financial transactions:
- Idempotency key required in request headers
- Existing transactions returned if key matches
- Prevents duplicate transaction processing
- Ensures consistent API behavior


6.3.4 Audit Logging

Comprehensive Audit Trail:

The platform implements comprehensive audit logging:
- User ID tracking for all operations
- Endpoint and method logging
- IP address and user agent capture
- Request body sanitization before logging
- Timestamp for all audit entries


6.4 Performance Standards

Performance Monitoring:

The platform implements performance monitoring:
- Response time tracking for all endpoints
- Performance metrics logged to database
- Automatic alerts for performance threshold violations (200ms threshold)
- Endpoint-level performance analysis
