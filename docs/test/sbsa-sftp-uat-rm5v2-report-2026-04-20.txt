Interpretation
================================================================================

The batch was rejected on the sub-batch limit (R500,000) before the per-
transaction limit check fired. Our file's batch total R500,003 exceeds the
sub-batch limit by R3, so SBSA rejected the file as a whole.

Because both profile limits are set to R500,000 (Cr Transaction Limit == Sub
Batch Limit), it is arithmetically impossible to build a test file that
exceeds the per-tx limit without also exceeding the sub-batch limit. In other
words, the "PART with one RJCT at 0006" path is not reachable with the current
TEST profile — any file containing a single tx > R500,000 will automatically
fail the batch-level check first.

Additional observation: Status Code 0009 is used by SBSA for two distinct
scenarios:
  • RM9 (2026-04-17): 0009 with no "RUN EXCEEDS LIMIT" description — mapped to
    invalid ordering account.
  • RM5v2 (2026-04-20): 0009 with description "RUN EXCEEDS LIMIT" — batch
    over the sub-batch limit.
Our pain002PollerService must therefore use the combination of the status
code AND the AddtlInf description to disambiguate the root cause, not the
code alone.

Questions to raise with SBSA
================================================================================

1. Can the TEST profile be re-configured so that Cr Transaction Limit is lower
   than the Sub Batch Limit (e.g. per-tx R100,000, batch R500,000)? That is
   the only way a Pain.001 can trip the per-tx limit (0006) without also
   tripping the sub-batch limit (0009).

2. Please confirm that Status Code 0009 is used for BOTH "invalid ordering
   account" (RM9 in 2026-04-17 UAT) and "RUN EXCEEDS LIMIT" (RM5v2 above), and
   that the AddtlInf Status Description is the authoritative differentiator.

3. Please confirm that SBSA does NOT issue a FINAUDTST when a file is
   rejected at INTAUDTST (GrpSts RJCT). This matches what we observed for
   RM9, RM10 and RM5v2 and is informally our assumption — we would like to
   bake it into our poller.

4. For our production go-live: if the production Cr Transaction Limit and Sub
   Batch Limit differ (which we expect and hope), we will need to know both
   values so we can correctly surface the right error to the operator when a
   customer submits a batch that hits either limit.
